Sizing Guide

Performance of a Logscpae system is dependent on the volume of data searched and the number of concurrent users. Performance relies heavily on fast disk i/o and a high enough core density on the server executing the search. Use the disk benchmarking tool to ensure that disk i/o is not slowing down searches and that there is adequate cpu utilization.

Memory

When sizing a Logscape system you need to consider two important parameters

  1. How much memory do I need for in-memory Indexes
  2. How much memory does the kernel need for caching

These parameters can be ignored for components that do not do any searching like the Forwarder. The amount of memory required for optimal performance equates roughly to twice your daily data ingestion volume. For an environment ingesting 20gb of data a day you would need a 40gb system. The memory should be alloted in the following way:

Indexing 30-40% memory Set -Xmx16gb in the boot.properties
Kernel Cache 50% memory freeKeep 20gb available to allow for file caching

This is the first place you should start when sizing a system. Take a look at this page to see how much memory your logscape components are taking.

[1] On the free download version of Logscpe the IndexStore can not run on a separate host from the Manager. Scaling out is disabled. In this case the sizing requirements for the Index Store also apply for the host running the Manager

[2] Swap usage - it is important that your memory usage does not enter swap. The swapiness kernel property is used to control when the kernel starts to use swap. It is recommended to switch swap off entirely and to size the system correctly.

Other Considerations

How the number of users affect the Manger - Keep a watch on the memory utilization of the aggspace and dashboard processes when the number of users of the system increases significantly. You may need to increase the heap of these processes ( see table ).

Large Alert Volumes - If you have hundreds of alerts in your deployment, watch the memory profile of the aggspace and allocate more memory when required.

Indexers - The typical deployment contains a Manager, search operations and data balancede across a few Index Stores and many Forwarders. Some environments make use of Indexers. If you have over 20 indexers in your environment increase the size of your aggspace heap.
